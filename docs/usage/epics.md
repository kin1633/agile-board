# エピック（案件）

**URL**: `/epics`

複数の Issue をまとめる大きな機能単位（エピック／案件）を管理します。Issue とエピックを紐付けることで、機能ごとの進捗や工数をまとめて把握できます。

---

## エピック一覧

登録済みのエピック一覧が表示されます。各エピックには以下が表示されます。

| 表示項目 | 説明 |
|---|---|
| タイトル | エピック名 |
| 説明 | エピックの概要（任意） |
| GitHub ステータスバッジ | GitHub Projects の Status フィールド値（例: `In Progress`）。同期後に自動設定される。github_status が存在する場合はアプリ手動ステータスは非表示 |
| GitHub 優先度バッジ | GitHub Projects の Priority フィールド値（例: `p0`）。フィールド定義の色を反映したバッジで表示される |
| リリース予定日 + 残り日数 | `due_date` が設定されている場合に表示。残り日数に応じて色が変わる |
| 着手日目安 | リリース予定日からリリースバッファと予定工数を逆算した目安日（青色表示）。詳細は[着手日目安の計算](#着手日目安の計算) |
| 着手日 | 実際の着手日（緑色表示）。手動設定または GitHub 同期時に自動設定 |
| 進捗バー | `closed` Issue 数 / 全 Issue 数のパーセンテージ |
| open Issue 数 | 未完了の Issue 件数 |
| 実績 / 予定 (h) | 配下の全 Task の工数サマリー（ワークログ合計 / `estimated_hours` 合計）。Task が未登録または工数が未入力の場合は非表示 |
| 推定スプリント数 | 残ポイント ÷ 平均ベロシティで算出した残りスプリント数 |
| 推定工数 | 残スプリント数 × チーム日次稼働時間 × 営業日数 |

---

## 工数サマリーの仕組み

```
Epic（案件）
  └─ Story Issue（epic_id で紐付け）
       └─ Task Issue（parent_issue_id で紐付け、GitHub Sub-issues）
            └─ estimated_hours / actual_hours（スプリント詳細で入力）
```

Epic の工数サマリーは **Task Issue の工数を Story → Epic の順に積み上げた合計値**です。

- Story に Task がない場合、またはすべての Task の工数が未入力の場合は工数サマリーは表示されません
- Story に紐付く Task が1件でも工数を持つ場合に表示されます

## 工数 CSV エクスポート

**URL**: `GET /epics/export?from=YYYY-MM-DD&to=YYYY-MM-DD`

エピック × 担当者 の工数をCSVファイルとしてダウンロードできます。

### 使い方

1. エピック一覧ページ（`/epics`）の右上で集計期間を入力する（デフォルト: 当月）
2. 「CSV エクスポート」ボタンをクリックするとファイルがダウンロードされる

### CSV フォーマット

```
案件名,担当者,予定工数(h),実績工数(h),予定工数(人日),実績工数(人日)
案件A,alice,10,8,1.43,1.14
```

### 集計ルール

| 項目 | 内容 |
|---|---|
| 集計粒度 | 案件（Epic）× 担当者 で1行 |
| 期間フィルタ | タスク（Sub-issue）の `closed_at` が指定期間内のものを集計 |
| 人日計算 | `工数(h) ÷ 1人日の基準時間`（設定画面の「一般設定」で変更可、デフォルト7h） |
| ファイル名 | `epics_export_YYYY-MM-DD_YYYY-MM-DD.csv` |

> Excel での文字化けを防ぐため CSV に UTF-8 BOM が付与されています。

## エピックの作成

「+ 新規エピック（案件）」ボタンをクリックするとフォームが展開します。

| 入力項目 | 必須 | 説明 |
|---|---|---|
| タイトル | ✅ | エピックの名前 |
| 説明 | - | エピックの概要・目的 |
| リリース予定日 | - | リリース目標日（`YYYY-MM-DD`） |
| 着手日 | - | 実際の着手日（`YYYY-MM-DD`）。GitHub 同期でも自動設定される |

> ステータス・優先度は GitHub Projects 同期によって自動設定されるため、作成フォームには含まれません。

---

## 着手日目安の計算

着手日目安（`estimated_start_date`）はサーバー側で計算された**表示専用**の値です。手動入力はできません。

```
開発完了目標日 = リリース予定日 - リリースバッファ日数（営業日）
着手日目安    = 開発完了目標日 - ceil(予定工数 / チーム日次工数)（営業日）
```

営業日の計算には**土日に加えて祝日管理に登録された祝日も除外**されます。

| 条件 | 挙動 |
|---|---|
| `due_date` 未設定 | 非表示 |
| 予定工数（Task 合計）が 0 | 非表示 |
| チームメンバー未登録（daily_hours 合計 = 0） | 非表示 |
| 上記条件を満たす場合 | 青色テキストで表示 |

> チーム日次工数はメンバー設定の `daily_hours` 合計値です。「設定 → メンバー」から各メンバーの稼働時間を設定してください。
>
> リリースバッファ日数は「設定 → 一般」から変更できます（デフォルト: 0日）。

---

## 着手日の自動設定

GitHub 同期後、以下の条件を満たすエピックには **着手日が自動設定**されます。

- `started_at` が未設定（`null`）
- エピックに紐付く Story Issue（`parent_issue_id IS NULL`）の中に GitHub Projects の **Status フィールドが "In Progress"** のものが1件以上ある

自動設定される日付は**同期実行日（`today()`）**です。

> `started_at` が既に設定されている場合は上書きされません。手動で修正したい場合は編集フォームから変更してください。

---

## エピックの編集

各エピックの「編集」ボタンから、タイトル・説明・リリース予定日・着手日を変更できます。

> ステータス・優先度は GitHub Projects 同期によって自動更新されるため、編集フォームには含まれません。

---

## エピックの削除

「削除」ボタンをクリックすると確認なしに削除されます。削除してもエピックに紐付いていた Issue が削除されることはありませんが、Issue の `epic_id` は `null` になります。

> 削除は取り消せません。ご注意ください。

---

## Issue とエピックの紐付け

Issue とエピックの紐付けは GitHub 同期では行われません。**スプリント詳細画面の Issue 一覧**から UI で設定できます。

### スプリント詳細からの紐付け手順

1. `/sprints/{id}` を開き「Issue 一覧」タブを表示する
2. 紐付けたい Issue 行のエピック選択ドロップダウンをクリックする
3. 一覧からエピック（案件）を選択すると即座に保存される
4. 紐付けを解除したい場合は「案件なし」を選択する

---

## 進捗バーの計算

```
進捗 % = closed な Issue 数 ÷ 全 Issue 数 × 100
```

Issue にストーリーポイントが設定されていない場合でも、件数ベースで進捗が計算されます。
